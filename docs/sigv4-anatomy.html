mincloud · internal/sigv4

SigV4 검증 해부

AWS CLI가 보낸 요청 한 개를 mincloud가 어떻게 "누가 보냈고, 변조되지 않았다"고 판정하는가 — 실제 캡처한 요청으로 따라가는 검증의 전 과정.

시크릿 키는 절대 네트워크로 전송되지 않는다.
클라이언트와 서버가 각자 같은 계산을 하고, 결과(서명)가 일치하는지만 본다.

등장인물: 키는 두 개, 역할은 다르다

"어떤 access key로 온 요청인지 어떻게 알까?" — 액세스키 ID는 비밀이 아니기 때문에 요청에 평문으로 실려 온다. 아이디/비밀번호 관계와 같다.

ACCESS KEY ID · 공개 식별자

MINCLOUDTESTKEY0000A

"누구인가"를 밝히는 아이디. Authorization 헤더의 Credential 첫 조각으로 평문 전송된다. 서버는 이 값으로 저장소에서 시크릿을 찾는다.

SECRET ACCESS KEY · 비밀

mincloud-test-secret-not-real

"본인인가"를 증명하는 비밀번호. 클라이언트와 서버 저장소에만 존재하고 절대 이동하지 않는다. 서명 계산의 재료로만 쓰인다.

실제 요청 해부

aws sts get-caller-identity 실행 시 CLI가 mincloud로 보낸 요청 원문. 색칠된 조각에 마우스를 올리거나 아래 범례를 눌러보세요.

POST / — captured 2026-07-05 20:28:14 KST
POST / HTTP/1.1
Host: localhost:9900
Content-Type: application/x-www-form-urlencoded; charset=utf-8
X-Amz-Date: 20260705T112814Z
Authorization: AWS4-HMAC-SHA256
    Credential=MINCLOUDTESTKEY0000A/20260705/ap-northeast-2/sts/aws4_request,
    SignedHeaders=content-type;host;x-amz-date,
    Signature=3e4361b978419419ae9c741d1c518fcf2930d8afdcaed1256962637320bb3984

Action=GetCallerIdentity&Version=2011-06-15   ← 본문이 곧 API 호출 (Query protocol)
Access Key ID누가 보냈나 — 서버가 시크릿을 찾는 조회 키
Credential Scope이 서명의 유효 범위 — 날짜 / 리전 / 서비스 / aws4_request. 범위 밖에서는 무효
SignedHeaders서명 계산에 포함된 헤더 목록 — 이 헤더들은 중간에서 바꾸면 서명이 깨진다
Signature클라이언트가 시크릿으로 계산한 HMAC 결과 — 서버가 재현해야 할 목표값

검증 파이프라인: 여섯 단계

각 단계는 실패 시 즉시 거부한다. 실제 AWS와 같은 에러 코드를 돌려주므로 CLI가 그대로 알아듣는다.

1

Authorization 헤더 파싱 ParseAuthorization

헤더에서 액세스키 ID · 스코프 · 서명된 헤더 목록 · 서명값 네 가지를 꺼낸다.

실패 → 403 IncompleteSignature
2

시크릿 조회 credstore.Lookup

액세스키 ID로 저장소에서 시크릿 키와 신원(Account · UserId · ARN)을 찾는다. 여기서 "누구인지"가 결정된다.

모르는 키 → 403 InvalidClientTokenId
3

Canonical Request 재구성 canonicalRequest

HMAC은 1바이트만 달라도 결과가 완전히 달라진다. 그래서 양쪽이 바이트 단위로 동일한 문자열을 만들도록 요청을 정규화한다. 마지막 줄의 본문 해시 덕분에 본문 변조도 잡힌다.

POST

4

String to Sign 조립

canonical request의 해시를 시각과 스코프로 한 번 더 감싼다. 다른 날짜 · 리전 · 서비스에서 서명을 재사용할 수 없게 만드는 단계.

AWS4-HMAC-SHA256
20260705T112814Z
20260705/ap-northeast-2/sts/aws4_request
SHA-256(canonical request)
5

서명 키 파생 + 서명 계산 signingKey

시크릿을 바로 쓰지 않고 4단 HMAC 체인으로 범위가 좁혀진 파생 키를 만든 뒤, 그 키로 String to Sign을 서명한다. (아래 체인 참고)

6

상수시간 비교 hmac.Equal

재계산한 서명과 헤더의 서명을 비교한다. 일반 == 비교는 "몇 번째 바이트에서 틀렸는지"가 응답 시간으로 새어 나가므로(타이밍 공격) 상수시간 비교를 쓴다. 일치하면 → 신원 XML 리턴.

불일치 → 403 SignatureDoesNotMatch

서명 키 파생 체인

최종 서명 키는 그날 · 그 리전 · 그 서비스에서만 유효한 파생 키다. 유출돼도 피해 반경이 좁고, 원본 시크릿은 역산할 수 없다.

HMAC( "AWS4" + 시크릿, "20260705" )→ kDate
↓
HMAC( kDate, "ap-northeast-2" )→ kRegion
↓
HMAC( kRegion, "sts" )→ kService
↓
HMAC( kService, "aws4_request" )→ kSigning
↓
HMAC( kSigning, String to Sign )→ Signature 3e4361b9…

이 설계가 막는 것들

도청

시크릿이 전송되지 않으므로 요청을 통째로 훔쳐봐도 비밀번호에 해당하는 것을 얻지 못한다.

변조 · 재사용

본문 해시가 서명에 들어가 내용을 바꾸면 깨지고, 시각·스코프가 들어가 다른 날짜/리전/서비스에서 재사용해도 무효다.

키 유출 반경

서명에 쓰이는 건 파생 키(kSigning)라, 유출돼도 그날 그 리전 그 서비스로 한정되고 원본 시크릿은 역산 불가.

테스트 전략: mincloud의 internal/sigv4 테스트는 자체 생성기가 아니라 실제 AWS CLI가 만든 서명 2개를 기준값으로 쓴다. 테스트 통과 = AWS 구현과 바이트 단위 일치라는 증명. 본문의 키/시크릿은 모두 테스트용 가짜 값이다.
