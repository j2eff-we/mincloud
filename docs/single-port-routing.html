포트 하나로 모든 AWS 서비스 — SigV4 스코프 라우팅

sts · iam · ec2를 서비스마다 포트 하나씩 열다가, nginx 경로 라우팅과 /etc/hosts 호스트 라우팅을 거쳐, 결국 요청이 이미 들고 있던 답을 발견하기까지 — 단일 포트 라우터의 설계 여정.

클라이언트는 서명하는 순간 목적지 서비스를 스스로 선언한다.
Credential=…/20260724/ap-northeast-2/ec2/aws4_request — 세 번째 조각이 곧 라우팅 키다.

문제: 서비스 하나 = 포트 하나?

처음 구조는 단순했다. sts는 9900, iam은 9910, ec2는 9920. 서비스가 셋일 때는 견딜 만하지만, 서비스를 하나 추가할 때마다 늘어나는 것들을 세어 보면 이야기가 달라진다.

.env — 서비스당 포트 시절의 클라이언트 설정
AWS_ENDPOINT_URL_STS=http://localhost:9900
AWS_ENDPOINT_URL_IAM=http://localhost:9910
AWS_ENDPOINT_URL_EC2=http://localhost:9920
# S3 붙이면 9930, DynamoDB 붙이면 9940, …

포트 번호 하나, 환경변수 하나, 서버 쪽 리스너 하나, 그리고 "이 서비스는 몇 번이었지?"라는 기억 부담 하나. S3, DynamoDB까지 가면 포트 지옥이다. 진짜 AWS는 서비스가 200개가 넘어도 클라이언트 설정은 리전 하나로 끝난다 — 우리도 엔드포인트 하나로 줄이고 싶다.

시도 1nginx 경로 라우팅 — 서명이 깨진다

흔한 리버스 프록시 패턴부터 떠올렸다. localhost/ec2로 받아서 프록시가 /ec2 프리픽스를 떼고 백엔드로 넘기는 구조. 웹 앱이라면 잘 동작하지만, SigV4 앞에서는 무너진다. SigV4는 URI 경로를 canonical request에 포함해 서명하기 때문이다.

canonical request — 두 번째 줄이 경로다 (sigv4-anatomy 참고)
POST
/ec2                    ← 클라이언트가 서명한 경로
(정렬된 쿼리스트링)
content-type:application/x-www-form-urlencoded; charset=utf-8
host:localhost
x-amz-date:20260724T031500Z
…
nginx가 /ec2 를 떼고 넘긴 뒤 — 서버의 재계산
POST
/                       ← 서버가 보는 경로. 1바이트만 달라도 HMAC은 전혀 다른 값
…
→ 403 SignatureDoesNotMatch

클라이언트는 /ec2를 넣고 서명했는데 서버는 /로 재계산하니 서명이 절대 일치할 수 없다. 그럼 프리픽스를 안 떼면? 이번엔 모든 서비스 핸들러가 /ec2 프리픽스를 알아야 하고, 결국 Go 안에서 경로를 보고 다시 라우팅해야 한다. 어차피 Go에서 라우팅할 거라면 nginx는 무의미하다.

시도 2/etc/hosts + 호스트 라우팅 — 동작하지만 셋업이 깨진다

그럼 진짜 AWS처럼 호스트네임으로 갈라보자. ec2.jeff.local, sts.jeff.local을 /etc/hosts에 등록하고 Host 헤더로 라우팅하는 방식이다.

동작한다

Host 헤더도 SignedHeaders에 들어가 서명되지만, 클라이언트와 서버가 같은 값을 본다 — 아무도 중간에서 바꾸지 않으니 서명은 깨지지 않는다. ec2.ap-northeast-2.amazonaws.com처럼 실제 AWS와 가장 비슷한 모양이기도 하다.

그러나

/etc/hosts 수정에는 sudo가 필요하고, 머신마다 다시 해줘야 한다. mincloud의 약속인 "클론하고 make run"이 여기서 깨진다. 코드 밖의 시스템 설정에 기대는 순간 재현 가능성이 사라진다.

해답: 답은 이미 요청 안에 있었다

Authorization 헤더를 다시 보자. SigV4는 서명의 유효 범위를 크리덴셜 스코프로 못 박는데, 그 안에 서비스 이름이 이미 들어 있다.

Authorization 헤더 — 세 번째 조각이 목적지
Authorization: AWS4-HMAC-SHA256
    Credential=MINCLOUDTESTKEY0000A/20260724/ap-northeast-2/ec2/aws4_request,
    SignedHeaders=content-type;host;x-amz-date,
    Signature=…

날짜 / 리전 / 서비스 / aws4_request — 서명 키 파생 체인에도 이 서비스 이름이 들어간다

이건 서버가 추측한 값이 아니다. 클라이언트가 서명하는 시점에 스스로 선언한 목적지이고, 다른 서비스용 서명은 그 서비스에서 무효가 되도록 설계된 값이다. 그렇다면 서버는 경로도 호스트도 볼 필요 없이, Authorization 헤더 하나만 파싱해서 디스패치하면 된다. 포트도, nginx도, /etc/hosts도 필요 없다.

1

Authorization 헤더 존재 확인

헤더가 아예 없으면 어느 서비스로 보낼지조차 알 수 없다.

없음 → 403 MissingAuthenticationToken
2

스코프 파싱 sigv4.ParseAuthorization

검증 파이프라인 1단계에서 쓰던 파서를 그대로 재사용한다. 파싱 결과 Authorization 구조체의 Service 필드가 라우팅 키.

파싱 실패 → 403 IncompleteSignature
3

테이블 조회 후 디스패치 services[auth.Service]

map[string]http.Handler에서 핸들러를 찾아 요청을 통째로 넘긴다. 서명 검증은 각 서비스 핸들러 안에 그대로 남는다 — 라우터는 봉투의 주소만 읽는 얇은 글루일 뿐, 봉인을 뜯지 않는다.

모르는 서비스 → 400 UnknownService

코드: 라우터 전체가 이게 전부다

핵심 분기 세 개, 에러 세 종류. 실제 AWS와 같은 XML 에러 포맷으로 응답하므로 aws CLI가 그대로 알아듣는다.

internal/router/router.go — New()
func New(services map[string]http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        authHeader := r.Header.Get("Authorization")
        if authHeader == "" {
            writeError(w, http.StatusForbidden, "MissingAuthenticationToken",
                "Request is missing Authentication Token")
            return
        }
        auth, err := sigv4.ParseAuthorization(authHeader)
        if err != nil {
            writeError(w, http.StatusForbidden, "IncompleteSignature", err.Error())
            return
        }
        h, ok := services[auth.Service]
        if !ok {
            writeError(w, http.StatusBadRequest, "UnknownService",
                "mincloud does not implement service '"+auth.Service+"'")
            return
        }
        h.ServeHTTP(w, r)
    })
}
cmd/mincloud/main.go — 조립은 테이블 하나
log.Fatal(http.Serve(ln, router.New(map[string]http.Handler{
    "sts": sts.Handler(store, *verbose),
    "iam": iam.Handler(store, *verbose),
    "ec2": ec2.Handler(store, *verbose),
})))

보너스: LocalStack의 4566이 이해된다

LocalStack이 수십 개 AWS 서비스를 단일 포트 4566으로 받는 이유가 바로 이 메커니즘이다. SigV4를 구현하는 순간 모든 요청이 목적지 라벨을 달고 오니, 단일 엔드포인트는 공짜로 따라온다. 그리고 클라이언트 설정은 이렇게 줄어든다.

.env — 단일 포트 이후
AWS_ENDPOINT_URL=http://localhost:9900   ← aws CLI v2의 글로벌 엔드포인트 환경변수. 이 한 줄이 전부

이 라우터는 프로세스 내 글루일 뿐이다. 서비스가 별도 프로세스로 분리되는 날이 오면 게이트웨이가 이 자리를 맡고, 서비스 핸들러는 그대로 이동한다. 남은 숙제 하나: S3의 presigned URL과 익명 요청은 Authorization 헤더가 없다 — 그때는 Host/경로 분기를 함께 쓸 예정이다. 본문의 캡처는 internal/router/router_test.go의 시나리오(스코프별 디스패치, UnknownService, MissingAuthenticationToken)와 동일하다.
