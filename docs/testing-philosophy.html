mincloud · testing

진짜 클라이언트가 오라클이다

AWS 호환 서버의 테스트 전략 — "우리끼리 맞다"는 회귀 테스트를 넘어, 진짜 AWS 도구가 만든 기준값과 진짜 aws CLI의 판정으로 호환성을 증명하는 세 계층.

호환성은 스스로 증명할 수 없다.
내가 만든 서명기로 내가 만든 검증기를 테스트하면, 둘이 같은 오해를 공유해도 통과한다. 기준값(oracle)은 반드시 진짜 AWS 도구에서 와야 한다.

L1캡처 벡터 유닛테스트 — AWS가 만든 정답지

internal/sigv4의 테스트는 자체 서명기가 아니라 실제 AWS CLI v2가 만든 서명 2개를 기준값으로 쓴다. 테스트용 가짜 키(MINCLOUDTESTKEY0000A)로 aws sts get-caller-identity를 두 번 실행해 캡처한 X-Amz-Date와 Signature 쌍이다.

internal/sigv4/sigv4_test.go — 캡처된 테스트 벡터
var capturedRequests = []struct {
	name      string
	amzDate   string
	signature string
}{
	{"first capture",  "20260705T112547Z", "285b60204ba0de27…306dab85"},
	{"second capture", "20260705T112814Z", "3e4361b978419419…20bb3984"},
}

TestVerifyCapturedRequests는 같은 요청을 재구성해 ComputeSignature가 캡처된 서명과 바이트 단위로 일치하는지 확인한다. 통과 = mincloud의 서명 계산이 AWS 구현과 정확히 같다는 증명. 네거티브도 같은 벡터로 검증한다 — 틀린 시크릿(TestVerifyRejectsWrongSecret), 본문 변조(TestVerifyRejectsTamperedBody) 모두 ErrSignatureMismatch여야 한다.

L2핸들러 유닛테스트 — 증명 위에 쌓는 자체 서명

검증기가 L1에서 이미 증명됐으므로, 핸들러 테스트는 매번 CLI를 부를 필요가 없다. sts·ec2 테스트의 signedRequest 헬퍼는 sigv4.ComputeSignature로 자체 서명한 요청을 만들어 핸들러에 넣는다 — 라우팅, 에러 코드, XML 구조에 집중하기 위해서다.

여기서 가장 재밌는 테스트는 잘못된 서비스 스코프 거부다. iam으로 서명한 요청을 sts 핸들러에 보내면, 액세스키와 시크릿이 전부 유효한데도 거부돼야 한다 — 서버가 자기 서비스 이름으로 서명을 재계산하므로 스코프가 다르면 서명이 어긋난다.

internal/service/sts/sts_test.go — TestRejectsWrongServiceScope
// Signed for iam, sent to the sts handler: must be rejected even though
// the access key and secret are valid.
r := signedRequest(t, "iam", body)

Handler(store, false).ServeHTTP(w, r)

// 기대: 403 + <Error><Code>SignatureDoesNotMatch</Code>…

ec2 쪽도 같은 방식이다 — TestRunThenDescribeInstances는 RunInstances가 pending 2대를 돌려주고 DescribeInstances에서 running으로 보이는 상태 전이를, TestErrorUsesEC2ResponseShape는 에러가 EC2 고유의 <Response> 형태(다른 서비스의 <ErrorResponse>가 아니라)로 나가는지를 확인한다.

L3e2e — 진짜 aws CLI가 최종 판정한다

scripts/e2e.sh는 서버를 실제로 빌드해 띄우고 진짜 aws CLI로 6가지를 체크한다. 통과 = 진짜 AWS 도구가 우리의 와이어 응답을 받아들인다는 뜻. aws CLI가 없는 환경에선 SKIP을 출력하고 성공으로 종료해 CI를 배려한다.

a

키 발급 → 재인증 라이프사이클

aws iam create-access-key --user-name alice로 새 키를 발급받고, 그 키로 다시 sts get-caller-identity를 호출해 arn:aws:iam::123456789012:user/alice가 돌아오는지 본다. 발급 → 저장 → 인증이 한 바퀴 도는 체크다.

b

네거티브: 틀린 시크릿 거부

AWS_SECRET_ACCESS_KEY=wrong-secret으로 호출이 성공하면 실패다. 긍정 체크만 있는 e2e는 "인증을 아예 안 해도" 통과할 수 있기 때문에, 거부가 되는지도 반드시 확인한다.

성공해버리면 → FAIL: request with wrong secret was accepted
$ ./scripts/e2e.sh
ok 1 - sts get-caller-identity returns account 123456789012
ok 2 - iam create-access-key issues a key for alice
ok 3 - new key authenticates as arn:...:user/alice
ok 4 - ec2 run-instances reports pending
ok 5 - ec2 describe-instances shows i-a328a6f918b16bbdc as running
ok 6 - wrong secret is rejected
PASS: 6 e2e checks

각 계층이 잡는 버그는 다르다

세 계층은 중복이 아니라 분업이다. 아래 계층이 증명한 것을 위 계층은 가정하고, 위 계층만 잡을 수 있는 버그에 집중한다.

L1 · 캡처 벡터

서명 알고리즘의 바이트 편차

정규화 순서, 인코딩, HMAC 체인 등 1바이트만 달라도 AWS와 어긋나는 종류의 버그. AWS가 만든 정답지와의 대조로만 잡힌다.

L2 · 핸들러

라우팅 · 에러 코드 · XML 회귀

잘못된 서비스로 요청이 새는 것, 에러 코드가 바뀌는 것, 응답 XML 구조가 깨지는 것 — 빠르고 촘촘하게 잡는다.

L3 · e2e

진짜 클라이언트만 아는 기대치

CLI의 파싱 실패, 우리가 몰랐던 필수 필드 누락 — 스펙 문서엔 없고 실제 클라이언트 코드에만 있는 기대를 어겼을 때 잡힌다.

다음 단계: 오라클을 하나 더 늘린다 — Terraform을 4번째 오라클로. aws CLI와는 다른 SDK 경로와 다른 기대치를 가진 클라이언트라, 같은 서버를 다른 각도에서 검증해 준다. 본문의 키/시크릿/서명은 모두 테스트용 가짜 값이며, 출처는 internal/sigv4/sigv4_test.go · internal/service/sts/sts_test.go · internal/service/ec2/ec2_test.go · scripts/e2e.sh.
