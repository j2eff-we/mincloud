mincloud · internal/service

Query 프로토콜 해부 — Action 하나로 움직이는 AWS API

aws CLI가 mincloud로 보내는 요청은 겉보기에 전부 똑같다. POST / 하나. 그런데 어떻게 IAM 키 발급도 되고 EC2 인스턴스 기동도 될까 — form 본문 한 줄이 API 전체를 지배하는 Query 프로토콜의 해부.

이 API는 REST가 아니다. URL 경로도, HTTP 메서드 구분도 없다.
모든 요청이 POST /이고, form-encoded 본문의 Action=...&Version=...이 곧 API 호출이다.

1. 라우터는 URL이 아니라 본문을 본다

REST라면 POST /instances, GET /users/jeff/keys 같은 경로가 있어야 한다. Query 프로토콜에는 없다. 서버가 하는 일은 본문을 url.ParseQuery로 풀고 Action 파라미터 하나로 switch 하는 것이 전부다. mincloud의 세 서비스 핸들러(iam, sts, ec2)는 모두 정확히 이 구조다.

internal/service/ec2/ec2.go — 핸들러의 뼈대 (iam, sts도 동일한 구조)
form, _ := url.ParseQuery(string(body))   // 본문 = 쿼리스트링 문법
action := form.Get("Action")

switch action {
case "RunInstances":       runInstances(w, reg, cred.Identity.Account, form)
case "DescribeInstances":  describeInstances(w, reg)
case "DescribeRegions":    describeRegions(w)
default:
    writeError(w, http.StatusBadRequest, "InvalidAction",
        "The action "+action+" is not valid for this web service.")
}

그럼 "어느 서비스로 온 요청인가"는 무엇이 구분할까? SigV4 서명의 credential scope다. 서명에 박힌 서비스 이름(.../ap-northeast-2/ec2/aws4_request)이 핸들러의 서비스와 다르면 service.Authenticate가 SignatureDoesNotMatch로 거부한다 — 실제 AWS와 같은 동작이다. SigV4 검증 해부 편에서 다룬 그 스코프가 여기서 라우팅 경계 역할까지 한다.

2. 실제 요청 해부: aws ec2 run-instances

aws ec2 run-instances --image-id ami-12345678 --instance-type t3.small 실행 시 CLI가 보내는 요청. 색칠된 조각에 마우스를 올리거나 아래 범례를 눌러보세요.

POST / — Content-Type: application/x-www-form-urlencoded
POST / HTTP/1.1
Host: localhost:9900
Content-Type: application/x-www-form-urlencoded; charset=utf-8
X-Amz-Date: 20260724T093012Z
Authorization: AWS4-HMAC-SHA256 Credential=AKIA.../20260724/ap-northeast-2/ec2/aws4_request, ...

Action=RunInstances&Version=2016-11-15&ImageId=ami-12345678&InstanceType=t3.small&MinCount=1&MaxCount=1
Action어떤 API를 부르는가 — REST의 메서드+경로에 해당. 서버 switch문의 분기 키
VersionAPI 모델의 버전 날짜 — 응답 XML 네임스페이스와 짝을 이룬다 (아래 3절)
파라미터전부 평탄한 key=value — JSON처럼 중첩 구조가 없다. 중첩·리스트는 이름 규칙으로 흉내낸다
수량 파라미터mincloud는 MinCount만큼 가짜 인스턴스를 만들고, InstanceType 생략 시 t3.micro로 채운다

리스트는 어떻게 보낼까? 배열 문법이 없으므로 번호 접미사로 평탄화한다. 이 직렬화 규칙까지가 프로토콜의 일부다.

리스트 직렬화 — 번호 붙은 접미사
SecurityGroupId.1=sg-aaa&SecurityGroupId.2=sg-bbb        ← EC2 스타일
Filter.1.Name=instance-type&Filter.1.Value.1=t3.small    ← 중첩도 점(.)으로 평탄화

한 가지 더 — 이 본문 전체가 SigV4 서명 대상이다. canonical request의 마지막 줄이 SHA-256(본문)이므로, Action이나 파라미터를 한 글자만 바꿔도 서명이 깨진다. Query 프로토콜과 SigV4는 이렇게 맞물려 있다.

3. 응답은 XML — 서비스마다 사투리가 다르다

요청 형식은 세 서비스가 똑같지만, 응답 XML로 오면 갈라진다. 먼저 네임스페이스: 서비스마다 다르고, 자세히 보면 버전이 곧 API 모델의 날짜다. 요청의 Version=과 응답 xmlns의 날짜가 짝을 이룬다.

서비스xmlns비고
IAMhttps://iam.amazonaws.com/doc/2010-05-08/2010년 모델을 아직 쓴다
STShttps://sts.amazonaws.com/doc/2011-06-15/https
EC2http://ec2.amazonaws.com/doc/2016-11-15/여기만 http — 이것도 바이트 단위로 맞춰야 한다

원소 명명 규칙도 갈라진다. IAM·STS는 UpperCamel, EC2는 lowerCamel에 리스트를 item으로 감싼다.

IAM CreateAccessKey — UpperCamel
<CreateAccessKeyResponse xmlns="...2010-05-08/">
  <CreateAccessKeyResult>
    <AccessKey>
      <UserName>jeff</UserName>
      <AccessKeyId>AKIA...</AccessKeyId>
      <Status>Active</Status>
      <SecretAccessKey>...</SecretAccessKey>
    </AccessKey>
  </CreateAccessKeyResult>
  <ResponseMetadata>
    <RequestId>...</RequestId>
  </ResponseMetadata>
</CreateAccessKeyResponse>
EC2 DescribeInstances — lowerCamel + item
<DescribeInstancesResponse xmlns="...2016-11-15/">
  <requestId>...</requestId>
  <reservationSet>
    <item>
      <reservationId>r-...</reservationId>
      <ownerId>123456789012</ownerId>
      <instancesSet>
        <item>
          <instanceId>i-...</instanceId>
          <instanceState>
            <code>16</code><name>running</name>
          </instanceState>
        </item>
      </instancesSet>
    </item>
  </reservationSet>
</DescribeInstancesResponse>

같은 프로토콜 가족인데 왜 이럴까? EC2가 가장 오래된 AWS API라 자기만의 관례가 굳어졌고, 이후 서비스들(IAM, STS)이 다른 관례로 표준화됐기 때문이다. 호환 구현 입장에서는 "Query 프로토콜 하나"가 아니라 사투리 두 개를 구현하는 셈이다. mincloud에서는 Go 구조체의 xml 태그가 이 사투리를 그대로 새긴다: iam.go는 xml:"CreateAccessKeyResult", ec2.go는 xml:"instancesSet>item".

4. 하이라이트: 에러 형식마저 두 갈래다

직접 구현하다 발견한 가장 큰 함정. 성공 응답만 다른 게 아니라 에러 봉투 자체가 다르다. IAM/STS는 ErrorResponse, EC2는 Response > Errors다 — 네임스페이스도 없고, RequestID의 대소문자까지 다르다 (RequestId vs RequestID).

IAM · STS — 잘못된 Action을 보내면
<ErrorResponse xmlns="...2010-05-08/">
  <Error>
    <Type>Sender</Type>
    <Code>InvalidAction</Code>
    <Message>Could not find operation ...</Message>
  </Error>
  <RequestId>...</RequestId>
</ErrorResponse>
EC2 — ImageId 없이 RunInstances 하면
<Response>                ← 네임스페이스 없음
  <Errors>
    <Error>               ← Type 없음
      <Code>MissingParameter</Code>
      <Message>The request must contain
        the parameter ImageId</Message>
    </Error>
  </Errors>
  <RequestID>...</RequestID>  ← ID가 대문자
</Response>

이걸 대충 한 형식으로 통일하면 어떻게 될까? CLI는 에러 코드와 메시지를 찾지 못해 정체불명의 에러를 뱉는다. "친절한 에러 메시지"조차 wire format을 정확히 맞춰야 전달된다. 그래서 mincloud는 errorResponse 구조체를 서비스 패키지마다 따로 둔다 — 공유하면 안 되는 것을 공유하지 않는 것도 설계다.

5. CLI가 속는 원리: botocore는 모델대로만 읽는다

aws CLI의 내부(botocore)는 XML을 "그냥" 파싱하지 않는다. 서비스마다 JSON으로 된 서비스 모델(어떤 Action이 있고, 응답의 어떤 원소가 어떤 필드인지)을 갖고 있고, 그 스키마에 적힌 이름의 원소만 골라 읽는다.

이름이 맞으면

모델의 필드로 채워져 "InstanceId": "i-..." 같은 JSON 출력으로 나온다. CLI는 상대가 진짜 AWS인지 알 방법도, 관심도 없다.

이름이 틀리면

에러가 아니라 침묵이다. 모델에 없는 원소는 그냥 무시되어 필드가 조용히 사라진다. instancesSet을 Instances로 쓰면 인스턴스 목록이 빈 채로 출력된다.

형식이 크게 틀리면

봉투(루트 원소, 에러 형식)가 어긋나면 그제서야 CLI가 파싱 에러를 낸다. 디버깅 난이도: 침묵 > 에러.

그래서 "aws CLI를 속인다"는 말의 실체는 마술이 아니다. 모델이 기대하는 wire format을 바이트 수준으로 재현하는 것, 그게 전부다. 거꾸로 말하면 botocore의 서비스 모델은 이 프로토콜의 가장 정확한 공개 명세서이고, mincloud의 진짜 테스트는 유닛 테스트가 아니라 실제 aws CLI가 정상 출력을 내는가이다.

검증: 진짜 CLI, 가짜 클라우드
$ aws ec2 run-instances --endpoint-url http://localhost:9900 \
    --image-id ami-12345678 --instance-type t3.small
{
    "ReservationId": "r-06d1c...",
    "Instances": [ { "InstanceId": "i-0f3ab...", "State": { "Code": 0, "Name": "pending" } } ]
}
# 응답은 pending으로 — 그리고 다음 describe-instances부터는 running.
# 실제 EC2의 상태 전이를 한 박자 흉내낸다.

구현 위치: 서비스별 핸들러는 internal/service/ 아래 패키지로 나뉜다 — iam/iam.go, sts/sts.go, ec2/ec2.go. 공통 동작(SigV4 인증 + 스코프 검사, XML 응답 프레이밍 service.WriteXML, AWS풍 UUID를 만드는 service.RequestID)은 internal/service/service.go에 있다. 응답 구조체는 encoding/xml 태그로 wire format을 선언하므로, 이 글의 XML 예시는 곧 코드의 구조체 정의와 1:1로 대응한다. 본문의 ID·키 값은 모두 예시용 가짜 값이다.
