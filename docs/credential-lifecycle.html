액세스 키의 일생 — 발급에서 인증까지

aws iam create-access-key 한 번으로 태어난 키가 어디에 저장되고, 어떻게 바로 다음 요청부터 "alice입니다"를 증명하는가 — 실제 e2e 실행 로그로 따라가는 자격 증명의 전체 수명.

시크릿 키가 네트워크를 건너는 순간은 일생에 단 한 번 — 발급 응답뿐이다.
그 후로는 클라이언트와 서버 저장소에만 존재하고, 서명 계산의 재료로만 쓰인다.

발급 해부: 키는 이렇게 태어난다

aws iam create-access-key --user-name alice 실행 시 mincloud가 돌려주는 응답 XML. 색칠된 조각에 아래 범례를 겹쳐 보세요. 이 SecretAccessKey 한 줄이 시크릿이 전송되는 처음이자 마지막 순간이다.

CreateAccessKeyResponse — internal/service/iam
<CreateAccessKeyResponse xmlns="https://iam.amazonaws.com/doc/2010-05-08/">
  <CreateAccessKeyResult>
    <AccessKey>
      <UserName>alice</UserName>
      <AccessKeyId>AKIA3QF0R7VH2M8XKZTB</AccessKeyId>
      <Status>Active</Status>
      <SecretAccessKey>u3kQ9rLxT2mYw8ZfVbN0aE5cH7jD1sGpKoR4tUiM</SecretAccessKey>
      <CreateDate>2026-07-24T02:12:10Z</CreateDate>
    </AccessKey>
  </CreateAccessKeyResult>
  <ResponseMetadata><RequestId>…</RequestId></ResponseMetadata>
</CreateAccessKeyResponse>
Access Key IDAKIA + 영숫자 16자 — 실제 AWS의 AKIA 접두사 관례 그대로. 이후 모든 요청에 평문으로 실려 오는 공개 식별자
Secret Access Key30바이트 난수를 base64로 인코딩해 정확히 40자 — 실제 AWS 시크릿도 40자다. 이 응답 이후 다시는 전송되지 않는다
UserName이 키가 증명할 신원의 이름 — UserID는 AIDA 접두사, ARN은 arn:aws:iam::123456789012:user/alice로 함께 생성된다
CreateDate발급 시각(UTC) — 2006-01-02T15:04:05Z 포맷
ACCESS KEY ID · 공개 식별자

"AKIA" + mustRandomAlphanumeric(16)

알파벳 대문자와 숫자(A–Z, 0–9)에서 crypto/rand로 16자를 뽑아 AKIA 뒤에 붙인다. 총 20자 — aws CLI가 아는 그 모양 그대로.

SECRET ACCESS KEY · 비밀

base64( rand.Read(30 bytes) )

30바이트는 base64로 정확히 40자가 되는 크기다. 패딩 없이 떨어지는 이 길이 맞춤은 우연이 아니라 실제 AWS 시크릿의 40자 규격을 겨냥한 것.

저장: 키 ID가 곧 조회 키다

응답을 보내기 직전, 서버는 새 키를 credstore에 넣는다. 구조는 단순하다 — 액세스키 ID를 키로, {시크릿, 신원} 을 값으로 하는 매핑.

internal/credstore — Put 직후의 상태
"AKIA3QF0R7VH2M8XKZTB" → Credential{
    SecretAccessKey: "u3kQ9rLxT2mYw8ZfVbN0aE5cH7jD1sGpKoR4tUiM",
    Identity: Identity{
        Account: "123456789012",
        UserID:  "AIDAW8K2NQ4TV0YJ6RXH",   ← "AIDA" + 영숫자 16자
        ARN:     "arn:aws:iam::123456789012:user/alice",
    },
}

이 구조가 AWS 콘솔의 그 유명한 경고 — "시크릿 키는 지금 이 화면에서만 볼 수 있습니다" — 를 설명한다. 서버 입장에서 시크릿의 용도는 단 하나, 들어온 요청의 서명을 재계산해 검증하는 것이다. 저장소에서 꺼내 사용자에게 다시 보여주는 경로는 애초에 만들지 않는 게 원칙이고, 그래서 발급 응답이 클라이언트가 시크릿을 손에 넣을 유일한 기회다. 잃어버리면 재발급뿐.

발급 즉시 살아있는 키

발급과 인증이 같은 저장소를 보기 때문에, 새 키는 응답이 도착한 순간부터 유효하다. 다음 요청을 이 키로 서명하면 무슨 일이 일어나는지 따라가 보자.

1

새 키로 서명해 요청

클라이언트가 AKIA3QF0…를 평문으로 Authorization 헤더에 싣고, 시크릿으로는 서명만 계산해 보낸다. 시크릿 자체는 두 번 다시 전송되지 않는다.

2

시크릿과 신원 조회 credstore.Lookup

서버가 헤더의 키 ID로 저장소를 찾으면 방금 Put한 시크릿과 Identity(Account · UserID · ARN)가 그대로 나온다.

3

SigV4 검증 → alice로 인증

찾은 시크릿으로 서명을 재계산해 비교하고, 일치하면 요청의 주인은 Lookup이 돌려준 신원 — 즉 alice다.

아래는 e2e 테스트(scripts/e2e.sh) 실행 중 캡처한 서버 로그 두 줄. jeff의 키로 alice의 키를 발급하고, 바로 다음 줄에서 그 키가 alice로 인증된다. 이 두 줄이 이 글의 전부다.

mincloud server log — captured during scripts/e2e.sh
11:12:10 iam CreateAccessKey by arn:aws:iam::123456789012:user/jeff
11:12:10 sts GetCallerIdentity by arn:aws:iam::123456789012:user/alice

직접 재현하기

진짜 aws CLI로 같은 장면을 만들 수 있다. 발급 응답에서 받은 키 쌍을 환경변수로 바꿔 끼우면 Arn이 alice로 바뀐다.

demo — aws CLI
$ source .env                                  # 개발용 키 + 엔드포인트
$ aws iam create-access-key --user-name alice  # 응답에서 키 쌍을 받는다

$ AWS_ACCESS_KEY_ID=AKIA3QF0R7VH2M8XKZTB \
  AWS_SECRET_ACCESS_KEY=u3kQ9rLxT2mYw8ZfVbN0aE5cH7jD1sGpKoR4tUiM \
  aws sts get-caller-identity
{
    "UserId": "AIDAW8K2NQ4TV0YJ6RXH",
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/alice"   ← 신원이 바뀌었다
}

단순화에 대한 고백: 지금의 mincloud에는 아직 CreateUser가 없다. CreateAccessKey가 처음 보는 UserName을 받으면 신원(UserID · ARN)을 즉석에서 함께 생성하는 상태다. 다음 마일스톤(Terraform을 위한 IAM CRUD)에서 유저 라이프사이클이 분리되면, 키 발급은 "이미 존재하는 유저"에 대해서만 가능해질 예정이다. 본문의 키/시크릿 값은 모두 예시용 가짜 값이다.
